# サイト内検索 設計ドキュメント

記事・ページを横断して検索できる `/search/` を追加する。

## 1. 目的

- 記事が増えても目的のコンテンツへすぐ辿り着けるようにする
- 静的サイト・リリースなし運用のまま検索を成立させる(外部 SaaS 不使用)

## 2. 方式の選定: Pagefind(astro-pagefind)

| 候補                             | 判断                                                                                               |
| -------------------------------- | -------------------------------------------------------------------------------------------------- |
| **Pagefind**(採用)               | ビルド成果物から静的インデックスを生成。サーバー不要・外部サービス不要。日本語の分かち書きにも対応 |
| Fuse.js 等のクライアント全文検索 | 全コンテンツを JS に同梱するためサイト成長でペイロードが肥大                                       |
| Algolia 等の SaaS                | 外部アカウント・API キーが必要(プロジェクト方針に反する)                                           |

- `astro-pagefind` integration(Astro ^7 対応)を使い、`astro build` 後に自動でインデックス生成
- 検索 UI も同パッケージの `Search` コンポーネント(Pagefind UI)を利用し、自前実装しない

## 3. 挙動

- `/search/` ページに検索ボックスを配置。入力時に `/pagefind/` 配下のインデックスを遅延ロード
- **インデックス対象は記事本文のみ**: 記事詳細の本文に `data-pagefind-body` を付与
  (Pagefind は `data-pagefind-body` が存在するサイトではその要素だけをインデックスする)
- Header ナビには追加せず、Articles 一覧から `/search/` への導線を置く
- Pagefind UI の配色は CSS 変数で上書きし、既存テーマ(`data-theme`)に追従させる

## 4. dev での確認

インデックスはビルド時に生成されるため、dev サーバーで検索を試すには一度 `bun run build` を
実行しておく(astro-pagefind が dev では `dist/pagefind` を配信する)。手順は README に記載。

## 5. テスト方針

- インデックス生成はビルド(CI の build ジョブ)で検証(`dist/pagefind/` の生成)
- UI は Pagefind 公式コンポーネントのため自前のユニットテストは追加しない
  (自作コンポーネントを足す場合は従来どおりテスト必須)

## 6. 非スコープ

- 検索結果のカスタム UI(必要になったら Pagefind の JS API で自作)
- タグ・works を含む横断検索のチューニング
